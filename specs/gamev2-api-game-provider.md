# gamev2 API Game Provider

## Context

`@battles/gamev2` currently runs only in stub mode: `MapLoader.tsx` fetches a hard-coded text map (`/maps/small-2p.txt`), parses it via `parseMap`, and hands a `StubGameProvider` (`packages/gamev2/src/providers/StubGameProvider.ts`) to the orchestrator. There is no path to load an existing game by id, no integration with `@battles/api`, and no multiplayer flow.

The legacy `@battles/game` application has a working API integration that supports remote multiplayer play — `packages/game/src/game/providers/api.ts`. We need to port the equivalent integration to gamev2 and, at the same time, consolidate the API client into a shared package so that `@battles/lobby`, `@battles/gamev2`, and (later) `@battles/game` all consume the same client.

### What the v1 API integration does

1. **`GameProvider` abstraction** — `packages/game/src/game/providers/base.ts:3` declares `get()`, `action(action)`, `wait(condition)`. Implementations: `LocalGameProvider`, `APIGameProvider`, `MockGameProvider`.
2. **`APIGameProvider`** — `packages/game/src/game/providers/api.ts:11`
   - Wraps `GameAPI` (axios) plus `PlayerActionLocalStorage` (per-game/per-user pending action cache).
   - `get()`: pulls `GameData`, pulls latest pending actions for this user (whichever of API record / local cache is newer), replays actions onto `latestMap`, returns `Game`.
   - `action(action)`: appends to local action cache. If `action.type === 'ready-player'`, pushes the cached action list to API via `PUT /game/:gameId/actions/:playerId`, then clears local cache. Always returns the locally-mutated `Game` — no roundtrip per non-ready action.
   - `wait()` is unimplemented; v1 polls inside `UiStore.onAllPlayersReady` (`packages/game/src/game/stores/ui.ts:148`) by calling `provider.get()` every 10s until `game.turn` advances.
3. **API endpoints** — `packages/api/src/main.ts`
   - `GET /game/_all` → `GameSummary[]`
   - `GET /game/:id` → `GameData`
   - `PUT /game` (body: `GameData`) — create
   - `GET /game/:id/view` → opaque view object
   - `PUT /game/:id/view` (body: viewData) — store view
   - `GET /game/:id/actions/:playerId` → `{ actions, updatedAt }`
   - `PUT /game/:id/actions/:playerId` (body: `Actions[]`) — server checks if all players have ready actions; if so, replays + appends resolved map, deletes pending actions, returns `{ resolved: true }`.
   - Server-side resolution means clients only need to poll `GET /game/:id` to detect `data.maps.length > previousTurn`.
4. **v1 wiring** — `packages/game/src/game/game.tsx:34` reads `?gameId&userId&local=true|false`. `local=true` → `LocalGameProvider`. Else → `APIGameProvider`. No `gameId` → `startExampleGame`.

### v1 vs v2 view data

| | v1 | v2 |
|---|---|---|
| Shape | `{ [territoryId]: { position: { x: number; y: number } } }` (`packages/game/src/game/stores/phaser.ts:26`) | text grid map string parsed by `parseMap()` → `RenderMap` (`packages/gamev2/src/map/MapParser.ts:21`) |
| Purpose | 2D Phaser screen positions per territory | Hex grid layout (`T`=territory, `g`=grass, `_`=empty) — territory IDs auto-generated by parser as `#1`, `#2`, … |
| Coupling to GameData | territory IDs in view must match GameData territory IDs | territory IDs in GameData must match parser-generated IDs |

The lobby already wraps both in a discriminated envelope (`packages/lobby/src/types.ts:10`):

```ts
type VersionedViewData =
  | { version: 'v1'; data: V1ViewData }
  | { version: 'v2'; data: string };

type StoredViewData = VersionedViewData | V1ViewData; // legacy: bare object, no version field
```

`NewGame.tsx:97` already writes the envelope on game creation; the API stores it opaquely. So envelope-aware reads are straightforward — the gamev2 client just has to assert `version === 'v2'` on load and surface a clear error otherwise.

### Lobby's existing API client

`packages/lobby/src/services/api.ts` already has a small fetch-based client (`createGame`, `listGames`) and the same hard-coded `BATTLES_API_HOSTNAME`. It will be replaced by the shared client below.

## Goals

1. Move the REST client into `@battles/api` so all browser packages consume one implementation.
2. Port `APIGameProvider` and `LocalGameProvider` into `@battles/gamev2`, wired through the orchestrator.
3. Detect attempts to load a v1 game in gamev2 and fail with a clear `console.error`.
4. Drop the duplicated client in `@battles/lobby`.

Non-goals: porting the v1 `@battles/game` axios client to use the shared client (left for a follow-up; legacy app is on its way out).

## Plan

### 1. Split `@battles/api` into client + server subpaths

The package currently exports only the lambda handler from `src/main.ts`. Browser packages can't import from it without dragging in `express`, `@aws-sdk/client-dynamodb`, and `dynamodb-onetable`. Solution: subpath exports.

```ts
// packages/api/src/client/viewData.ts
import type { StoredViewData, V1ViewData, VersionedViewData } from './types';

export function isV1ViewData(stored: StoredViewData): stored is V1ViewData | { version: 'v1'; data: V1ViewData } {
  if (stored && typeof stored === 'object' && !('version' in stored)) return true;
  return (stored as VersionedViewData).version === 'v1';
}

/** Returns the v2 map text. Throws if the stored view data is v1 (caller logs). */
export function unwrapV2MapText(stored: StoredViewData): string {
  if (isV1ViewData(stored)) throw new Error('v1-view-data');
  const env = stored as VersionedViewData;
  if (env.version !== 'v2') throw new Error(`unknown-view-data-version:${(env as any).version}`);
  return env.data;
}
```

```ts
// packages/api/src/client/GameApiClient.ts
import type { Actions, GameData } from '@battles/models';
import type { GameSummary, PlayerActionRecord, StoredViewData, VersionedViewData } from './types';

export const BATTLES_API_HOSTNAME = 'https://dctg63fuac.execute-api.ap-southeast-2.amazonaws.com';

export class GameApiClient {
  constructor(private endpoint: string = BATTLES_API_HOSTNAME) {}

  async listGames(): Promise<GameSummary[]> {
    const res = await fetch(`${this.endpoint}/game/_all`);
    if (!res.ok) throw new Error(`listGames failed: ${res.status}`);
    return res.json();
  }

  async getGameData(gameId: string): Promise<GameData> {
    const res = await fetch(`${this.endpoint}/game/${gameId}`);
    if (!res.ok) throw new Error(`getGameData failed: ${res.status}`);
    return res.json();
  }

  async createGame(gameData: GameData, viewData: VersionedViewData): Promise<void> {
    const g = await fetch(`${this.endpoint}/game`, {
      method: 'PUT',
      headers: { 'Content-Type': 'application/json' },
      body: JSON.stringify(gameData),
    });
    if (!g.ok) throw new Error(`createGame (game) failed: ${g.status}`);

    const v = await fetch(`${this.endpoint}/game/${gameData.id}/view`, {
      method: 'PUT',
      headers: { 'Content-Type': 'application/json' },
      body: JSON.stringify(viewData),
    });
    if (!v.ok) throw new Error(`createGame (view) failed: ${v.status}`);
  }

  async getViewData(gameId: string): Promise<StoredViewData> {
    const res = await fetch(`${this.endpoint}/game/${gameId}/view`);
    if (!res.ok) throw new Error(`getViewData failed: ${res.status}`);
    return res.json();
  }

  async getPlayerActions(gameId: string, playerId: string): Promise<PlayerActionRecord> {
    const res = await fetch(`${this.endpoint}/game/${gameId}/actions/${encodeURIComponent(playerId)}`);
    if (res.status === 404) return { actions: [], updatedAt: 0 };
    if (!res.ok) throw new Error(`getPlayerActions failed: ${res.status}`);
    return res.json();
  }

  async putPlayerActions(
    gameId: string,
    playerId: string,
    actions: Actions.ModelAction[],
  ): Promise<{ resolved: boolean }> {
    const res = await fetch(`${this.endpoint}/game/${gameId}/actions/${encodeURIComponent(playerId)}`, {
      method: 'PUT',
      headers: { 'Content-Type': 'application/json' },
      body: JSON.stringify(actions),
    });
    if (!res.ok) throw new Error(`putPlayerActions failed: ${res.status}`);
    return res.json();
  }
}
```

#### `packages/gamev2/src/providers/APIGameProvider.ts`

Direct port of v1 logic, using `GameApiClient`:

```ts
import { Game, GameMap } from '@battles/models';
import type { Actions, GameData } from '@battles/models';
import { GameApiClient, unwrapV2MapText } from '@battles/api/client';
import type { GameProvider } from './GameProvider';
import { PlayerActionStorage } from './PlayerActionStorage';

export class APIGameProvider implements GameProvider {
  private readonly storage: PlayerActionStorage;
  private readonly api: GameApiClient;
  private cachedGameData: GameData | null = null;

  constructor(private readonly gameId: string, private readonly userId: string, api?: GameApiClient) {
    this.api = api ?? new GameApiClient();
    this.storage = new PlayerActionStorage(gameId, userId);
  }

  async get(): Promise<Game> {
    this.cachedGameData = await this.api.getGameData(this.gameId);
    const actions = await this.findLatestActions();
    const game = new Game(this.cachedGameData);
    const map = new GameMap(game.latestMap);
    for (const action of actions) map.applyAction(action);
    return game;
  }

  async action(action: Actions.ModelAction): Promise<Game> {
    this.storage.addAction(action);
    if (action.type === 'ready-player') await this.pushLatestActions();
    if (!this.cachedGameData) throw new Error('action() called before get()');
    const game = new Game(this.cachedGameData);
    const map = new GameMap(game.latestMap);
    map.applyAction(action);
    return game;
  }

  async getMapText(): Promise<string> {
    const stored = await this.api.getViewData(this.gameId);
    try {
      return unwrapV2MapText(stored);
    } catch (e) {
      console.error(`[APIGameProvider] Cannot load gameId=${this.gameId} in gamev2: view data is not v2`, e);
      throw e;
    }
  }

  private async pushLatestActions(): Promise<void> {
    const actions = this.storage.currentActions.actions;
    const ready = actions.find((a) => a.type === 'ready-player') as Actions.ReadyPlayerModelAction | undefined;
    if (!ready) throw new Error('Pushing actions without a ready action');
    await this.api.putPlayerActions(this.gameId, ready.playerId, actions);
    this.storage.saveActions([]);
  }

  private async findLatestActions(): Promise<Actions.ModelAction[]> {
    const apiRecord = await this.api.getPlayerActions(this.gameId, this.userId);
    const localRecord = this.storage.currentActions;
    if (apiRecord.updatedAt > localRecord.updatedAt) {
      this.storage.saveActions(apiRecord.actions);
      return apiRecord.actions;
    }
    return localRecord.actions;
  }
}
```

#### `packages/gamev2/src/providers/LocalGameProvider.ts`

Reads/writes via the same localStorage layout the lobby uses (`packages/lobby/src/services/local-storage.ts`) so saved games created in the lobby are loadable. v1's `LocalGameProvider` resolves the turn synchronously when all players ready (hot-seat). Match that.

```ts
import { Game, GameMap } from '@battles/models';
import type { Actions, GameData } from '@battles/models';
import { unwrapV2MapText, type StoredViewData } from '@battles/api/client';
import type { GameProvider } from './GameProvider';

type SavedGame = { gameData: GameData; viewData: StoredViewData; lastUpdated: number };

const KEY_PREFIX = 'graph-battles-';

export class LocalGameProvider implements GameProvider {
  constructor(private readonly gameId: string, _userId: string) {}

  async get(): Promise<Game> {
    return new Game(this.load().gameData);
  }

  async action(action: Actions.ModelAction): Promise<Game> {
    const saved = this.load();
    const game = new Game(saved.gameData);
    const map = new GameMap(game.latestMap);
    map.applyAction(action);
    if (action.type === 'ready-player' && map.players.every((p) => p.ready)) {
      game.resolveTurn();
    }
    saved.gameData = game.data;
    saved.lastUpdated = Date.now();
    this.save(saved);
    return game;
  }

  async getMapText(): Promise<string> {
    try {
      return unwrapV2MapText(this.load().viewData);
    } catch (e) {
      console.error(`[LocalGameProvider] Cannot load gameId=${this.gameId} in gamev2: view data is not v2`, e);
      throw e;
    }
  }

  private load(): SavedGame {
    const raw = window.localStorage.getItem(KEY_PREFIX + this.gameId);
    if (!raw) throw new Error(`Game '${this.gameId}' not in localStorage`);
    return JSON.parse(raw) as SavedGame;
  }

  private save(saved: SavedGame): void {
    window.localStorage.setItem(KEY_PREFIX + this.gameId, JSON.stringify(saved));
  }
}
```

### 5. Orchestrator wiring

Currently `GameOrchestrator` mutates `map` directly and never calls `provider.action()`. To make multiplayer work, every user action must flow through the provider.

Add `await this.provider.action({...})` calls in:

| Method | Action emitted |
|---|---|
| `onTerritoryAction` | `{ type: 'territory', territoryId, action }` |
| `onCancelTerritoryAction` | `{ type: 'territory', territoryId, action: null }` |
| `moveSelectedUnits` | `{ type: 'move-units', unitIds, destinationId }` |
| `onCancelMove` | `{ type: 'move-units', unitIds, destinationId: null }` |
| `onReadyPlayer` | `{ type: 'ready-player', playerId, isReady: true }` |

Provider returns the locally-mutated `Game`; orchestrator continues to trust its in-memory `map` (already mutated). Only `ready-player` round-trips to the API.

#### Filter the player cycle to this user (remote only)

v1 calls `setFilteredUserIds([userId])` (`packages/game/src/game/game.tsx:61`). Equivalent for v2:

- Add `userId?: string` to `StoreState`.
- In `GameOrchestrator.initialise`, derive `playableplayerIds` for the cycle from `game.users.find(u => u.id === userId)?.playerIds ?? map.playerIds`.
- Use `playableplayerIds` instead of `map.playerIds` in the `onReadyPlayer` cycle so a remote tab only readies its own user's players.

#### Replace local resolution with poll on remote

For `LocalGameProvider`, keep the current `onAllPlayersReady` → `startResolution` flow (hot-seat).

For `APIGameProvider`, add a `'waiting'` value to `turnPhase` (`packages/gamev2/src/state/types.ts:44`). When the user readies their last player:

```ts
private async onAllPlayersReady(): Promise<void> {
  if (this.isLocalProvider) {
    // existing path
    await this.startResolution(...);
    return;
  }
  this.store.setState({ turnPhase: 'waiting' });
  this.startPollingForResolvedTurn();
}

private startPollingForResolvedTurn(): void {
  const startTurn = this.store.getState().turn;
  const tick = async () => {
    if (this.pollAborted) return;
    const game = await this.provider.get();
    if (game.turn > startTurn) {
      this.store.setState({ game, map: new GameMap(game.latestMap), turn: game.turn, turnPhase: 'next-player', ... });
      return;
    }
    this.pollTimeout = setTimeout(tick, 10_000);
  };
  tick();
}
```

10s cadence matches v1. Cancel on `dispose()` and on user-driven turn change.

## Risks

- **Subpath exports + bundlers**: Vite (lobby, gamev2) and esbuild (api server) both honour the `exports` field. Need to verify tsc emits `client/index.d.ts` next to the JS — tsconfig already has `outDir: ./dist/tsc/` and `declaration: true`, so layout should mirror `src/`.
- **Local action key prefix collision**: same `gameId` could exist in both v1 and v2 caches. Fixed by using `graph-battles-v2-actions-` for the v2 prefix.
- **Polling**: only one poll loop per orchestrator; must cancel on dispose / replay / next-turn to avoid duplicate fires after StrictMode remount.
- **Server action submission ordering**: v1 only ever pushes the cached action list on `ready-player`, so the server sees a single PUT per turn per player. We retain that behaviour — no need to debounce or batch.
- **v1 game opening in gamev2**: detection is by view-data shape (no `version` field, or `version: 'v1'`). If a future v3 ever lands, `unwrapV2MapText` already throws on unknown versions.

## Open questions

- Do we want to render an inline error UI for v1 games, or just leave the `console.error` and a blank screen? Spec assumes a simple error message in `MapLoader`.
- Lobby v2 view text maps for 2/3/4 players: who authors them? Tracked as a follow-up.
